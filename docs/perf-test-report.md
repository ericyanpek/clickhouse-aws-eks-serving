# ClickHouse on EKS —— 性能测试报告

> 测试日期:2026-07-05
> 数据集:ClickBench `hits`(Yandex Metrica 网页分析,99,997,497 行,单张 105 列宽表)
> 结论先行:**在 2×i8g.4xlarge(本地 NVMe)的 1 分片 × 2 副本拓扑上,43 条 ClickBench 查询 best-of-3 总耗时 18.76s、平均 0.436s、中位数 0.175s;读 QPS 随查询重量差 1–2 个数量级(全表扫聚合 ~48、命中索引点查 ~2,780),1→2 并发接近完美线性(2.0×);重扫描查询靠 `parallel_replicas` 可再提速最高 1.8×;单副本故障时服务不中断、QPS 降约 50%、副本约 65s 自愈。**

## 2. 阶段一:单节点查询延迟(43 条 ClickBench 查询)

每条查询跑 3 次取最优(ClickBench 标准的热运行口径),单连接(并发=1)。

### 汇总

| 指标 | 值 |
|---|---|
| 查询总数 | 43 |
| 总耗时(best 累加) | **18.76s** |
| 平均 | 0.436s |
| 中位数(p50) | 0.175s |
| p90 | 1.268s |
| 最快(Q1 `COUNT(*)`) | 0.002s |
| 最慢(Q29) | 5.211s |

### 分布特征

- **约 65% 的查询在亚秒级**(<0.3s),点查/前导列过滤类查询命中稀疏索引,毫秒级返回。
- **少数扫描密集型查询是单节点算力天花板的标志**(见下),它们扫描大范围列 + 高基数聚合,受限于单节点 CPU。

### 最慢的 5 条(单节点扫描天花板)

| 查询 | best(s) | 特征 |
|---|---|---|
| Q29 | 5.211 | 大范围 URL/正则相关重聚合 |
| Q19 | 1.665 | 高基数 GROUP BY |
| Q24 | 1.347 | 多列扫描聚合 |
| Q34 | 1.284 | 宽扫描 + 排序 |
| Q35 | 1.268 | 宽扫描 + 排序 |

> 完整逐条数据见 [`docs/perf-results/clickbench-43queries.csv`](./perf-results/clickbench-43queries.csv)。

## 3. 阶段二:parallel_replicas 单查询加速(重查询)

在最慢的查询上开启 `parallel_replicas`(2 副本并行扫同一份数据),验证「分片前先用 parallel_replicas 顶」的设计主张。

设置:`allow_experimental_parallel_reading_from_replicas=2, max_parallel_replicas=2, cluster_for_parallel_replicas='main'`

| 查询 | 关闭(s) | 开启(s) | 效果 |
|---|---|---|---|
| **Q24** | 1.337 | 0.737 | **1.81× 加速** ✅ |
| **Q29**(最重) | 5.269 | 3.958 | **1.33× 加速** ✅ |
| Q19 | 1.778 | 2.489 | 0.71×(变慢)⚠️ |
| Q33 | 1.232 | 2.669 | 0.46×(变慢)⚠️ |
| Q34 | 1.392 | 4.627 | 0.30×(变慢)⚠️ |

### 结论(诚实,不是每条都加速)

- **对大扫描/聚合查询(Q24/Q29)有效**:2 副本分摊扫描量,单查询提速 1.3–1.8×。这验证了在撞到「单查询算力天花板」时,`parallel_replicas` 能把分片的引入再往后推。
- **对轻量 / 带 LIMIT / 协调开销敏感的查询反而变慢**:并行协调的固定开销 > 扫描收益。
- **实践建议**:`parallel_replicas` 应**按查询选择性开启**(针对已知的重扫描查询),而非全局默认打开。

## 4. 阶段三:读并发 / QPS 扩展

> ⚠️ **先厘清一个关键认知**:QPS 不是集群的固定指标,而是**查询复杂度的函数**。本节 4.1 用一条「全表扫描聚合」的重查询测**副本读扩展的线性度**(所以 QPS 数值很低,约几十);4.2 补测了不同重量查询的真实 QPS(点查可达数千)。**不要把 4.1 的 ~52 QPS 当成集群的吞吐上限**——它是最重查询的下限。

### 4.1 读扩展线性度(用全表扫描重查询)

从独立的 bench 节点用 `clickhouse-benchmark` 打 ClusterIP service(轮询 2 副本),查询为全表 `GROUP BY RegionID`(**每次扫全部 1 亿行**的重聚合),并发阶梯 1→16,每档稳态 QPS:

| 并发 | 稳态 QPS | 相对 c=1 | 说明 |
|---|---|---|---|
| 1 | ~22.5 | 1.0× | 单连接基线 |
| 2 | ~44.8 | **2.0×** | **接近完美线性 —— 2 副本读扩展生效** |
| 4 | ~50.5 | 2.2× | 开始饱和 |
| 8 | ~52 | 2.3× | 饱和 |
| 16 | ~52 | 2.3× | 饱和(无进一步提升) |

峰值扫描吞吐:**~50 亿行/秒、~20 GiB/s**(RPS/带宽,本地 NVMe + 向量化扫描)。

**读扩展结论:**

- **1→2 并发接近完美线性(2.0×)**:验证读 QPS 随副本数线性扩展,ClusterIP 轮询把负载均匀打到 2 副本。
- **4 并发后饱和于 ~52 QPS**:因为测试查询是全表扫描聚合(每查询扫 1 亿行),**2 台 i8g 的 CPU 在约 4 并发即吃满**——这正是「读吞吐上限 = 副本数 × 单节点并发能力」的体现。对这种重扫描负载,提升 QPS 上限的手段是**加副本**(线性)。
- 峰值扫描吞吐 **~50 亿行/秒、~20 GiB/s**:从「每秒扫描行数(RPS)」看,2 台 i8g 的扫描能力非常强——**QPS 只有 52 是因为每个查询扫的行太多,不是集群弱**。
- 外推:3 副本时该饱和 QPS 预期约 ~75(线性外推)。

### 4.2 不同查询类型的真实 QPS(并发 8)

同一集群,QPS 随查询重量差 1–2 个数量级。真实 BI/serving 负载几乎都带过滤(时间范围、维度),命中 ClickHouse 稀疏主键索引(本表主键 `CounterID, EventDate, ...`),只扫一小部分数据,QPS 远高于全表扫描。实测:

| 查询类型 | QPS | RPS(每秒扫描行数) | 说明 |
|---|---|---|---|
| **纯连接开销**(`SELECT 1`,无扫描) | **~5,860** | — | 集群请求处理上限 |
| **点查**(`WHERE CounterID=?`,命中主键索引) | **~2,780** | 4,400 万 | 只扫该 key 的数据块 —— 比全表扫高 ~58× |
| **带过滤的聚合**(命中索引后 GROUP BY,典型 BI) | **~370** | 31.5 亿 | 生产 serving 负载的典型区间 |
| **全表聚合**(§4.1 的重查询,对照) | **~48** | 48 亿 | 每次扫全部 1 亿行 |

> 原始日志:[`docs/perf-results/qps-by-query-type.txt`](./perf-results/qps-by-query-type.txt)

**关键结论:**

1. **QPS 由查询复杂度决定,不是集群固定属性**:同一集群,查询从「全表扫」变「命中索引」,QPS 从 ~48 跳到 ~2,780(**58×**)。
2. **看 RPS(每秒扫描行数)更能反映硬件真实能力**:全表聚合虽只 48 QPS,但 RPS 达 48 亿行/秒——扫描吞吐极强,只是单查询「活儿」大。
3. **真实 serving/BI 负载落在点查~带过滤聚合区间(几百到数千 QPS)**,而非全表扫的几十。选型/容量规划应以**目标查询的实际形态**测 QPS,不能用全表扫的数字代表集群能力。

## 5. 阶段四:HA 混沌测试(读负载下杀副本)

在持续读负载(并发 4,`--continue_on_errors`)下,杀掉一个副本 Pod `chi-ch-main-0-1-0`,观察服务连续性与恢复。

| 阶段 | QPS | 错误 | 服务状态 |
|---|---|---|---|
| 杀之前(2 副本) | ~47–48 | 0 | 正常 |
| **杀瞬间** | — | **仅 1 个**(in-flight 查询) | 不中断 |
| 杀之后(1 副本) | ~25–26 | 0(后续) | **持续可用** |
| 恢复后 | 回升 | 0 | 2 副本 |

- **副本自愈时间:约 65 秒**(Pod 重建 + 本地 NVMe 上数据仍在,快速恢复)。
- 注:本次是 **Pod 被杀、节点存活**,本地 NVMe 数据卷还在,故恢复极快。若是**节点整体丢失**(本地盘随之丢失),则需从存活副本全量重建(分钟级,取决于数据量),这是本地 NVMe 方案的固有取舍。

### 结论

完整验证了 HA 设计:**单副本故障 → QPS 降约 50%(2→1),服务持续可用,仅损失 1 个瞬时查询,约 65s 自愈**。符合「多主对等复制、无主从选举、挂一个其余照常」的 ReplicatedMergeTree 特性。

## 6. 总体结论

| 维度 | 结果 | 印证的设计主张 |
|---|---|---|
| 单节点查询性能 | 43 查询平均 0.436s,65% 亚秒 | i8g + 本地 NVMe + 向量化扫描,宽表 OLAP 甜点区 |
| 压缩效率 | 3.8× | 列存 + 压缩 |
| 单查询天花板 | 最慢 5.2s(Q29),重扫描受单节点 CPU 限 | 「scale-up first」——单节点能力有天花板 |
| parallel_replicas | 重查询 1.3–1.8× 加速(轻查询反而慢) | 「分片前先用 parallel_replicas 顶」——对重查询成立,需选择性开启 |
| 读扩展 | 1→2 并发 2.0× 线性 | 「加副本 = 线性读扩展」 |
| QPS(随查询重量) | 全表扫 ~48 → 带过滤聚合 ~370 → 点查 ~2,780 → 空查询 ~5,860 | QPS 是查询复杂度的函数,非集群固定值 |
| 读吞吐上限 | 重扫描负载下 ~4 并发饱和于单节点 CPU;峰值 ~50 亿行/s 扫描 | 「读吞吐 = 副本数 × 单节点并发」 |
| HA | 单副本故障不中断,QPS 降 50%,~65s 自愈 | 「多主对等,挂一个其余照常」 |

### 与设计定位的呼应

本轮压测在真实 AWS 环境上验证了本方案的核心主张(详见 [`docs/community-corroboration.md`](./community-corroboration.md)):**1 分片垂直做大、本地 NVMe 追求 IO 上限、加副本做线性读扩展、重查询用 parallel_replicas 推迟分片**。所有数字均为实测,`parallel_replicas` 对轻查询变慢等**不利结果也如实记录**,未做选择性呈现。

### 后续可深化(本轮未覆盖)

- **写入 / merge 吞吐压测**(用 NYC Taxi 数据集)——本轮聚焦读,未压写。
- **3 副本口径复测**——当前 2 副本(i8g 容量所限),补齐第 3 副本后可验证 QPS 线性外推。
- **节点级故障恢复计时**——本轮是 Pod 级(数据卷存活);节点整体丢失 + 本地盘丢失的全量重建时间未测。
